# MenuLens — Scoring Algorithm Design

## Overview

Dish recommendations are computed using a **content-based filtering** approach. Each dish is represented as a feature vector, each user maintains a preference profile, and scoring is a weighted combination of flavor similarity and ingredient history signals.

---

## Dish Data Structure

Dishes are stored in two layers: a **global entry** for the flavor profile (shared across all restaurants) and a **restaurant-specific entry** for display data.

### Global Entry

Sourced once, cached, and reused whenever the same dish appears at any restaurant.

```json
{
  "dish_key": "pad_thai",
  "flavor_vector": {
    "umami":    7,
    "salty":    6,
    "sweet":    5,
    "bitter":   1,
    "sour":     7,
    "spicy":    4,
    "richness": 4
  },
  "base_ingredients": ["rice noodles", "egg", "tamarind", "peanut"],
  "source":     "lookup | llm | user_verified",
  "confidence": 0.0–1.0
}
```

### Restaurant-Specific Entry

Layered on top of the global entry. Contains display data and any restaurant-specific ingredient overrides.

```json
{
  "dish_key":      "pad_thai",
  "restaurant_id": "uuid",
  "display_name":  "Pad Thai",
  "description":   "Wok-tossed rice noodles with shrimp, egg, bean sprouts",
  "price":         "$14.50",
  "ingredients":   ["shrimp", "rice noodles", "egg", "bean sprouts", "peanut"],
  "section":       "Mains"
}
```

> **Note:** When a restaurant-specific ingredient list exists, it overrides the global `base_ingredients` for scoring purposes. The restaurant may use different proteins or preparation than the canonical version of the dish.

---

## Flavor Vector Sourcing

Flavor vectors are populated via a **hybrid lookup strategy**, in order of priority:

| Step | Source | Action |
|---|---|---|
| 1 | Restaurant-specific entry | Use if exists |
| 2 | Global entry (lookup table) | Use if exists |
| 3 | LLM background job | Generate once, store as global |
| 4 | Human verification | Review and confirm LLM output |

The LLM generation happens **offline in the background**, never during a menu scan. This keeps it out of the critical user-facing path.

### Source and Confidence

Every global flavor vector carries a `source` and `confidence` field that tracks how much to trust it.

**Source values:**

- `lookup` — manually maintained table of common dishes. Written intentionally by a human. Highest trust.
- `llm` — generated by the LLM background job the first time this dish was encountered. Has not been reviewed.
- `user_verified` — a human has reviewed and confirmed or corrected the LLM output.

**Confidence values:**

| Source | Confidence |
|---|---|
| `lookup` | 1.0 |
| `user_verified` | 1.0 |
| `llm` | 0.7 |

Confidence is used at scoring time to scale the flavor signal. The less trusted the vector, the more the score falls back to the coarser cuisine-level affinity:

```python
effective_flavor_score = (
    cosine_similarity(dish.flavor_vector, user_preference_vector) * dish.confidence
    + cuisine_affinity * (1 - dish.confidence)
)
```

**Confidence degradation:** If the same dish at the same restaurant is re-scanned and its description has changed significantly, confidence is multiplied by 0.85 to flag it for re-generation without immediately invalidating it.

## User Preference Profile

The user profile is organized **per cuisine × per section** (e.g. Korean mains, Italian desserts). This captures the fact that a user may prefer different flavor intensities within the same cuisine depending on the course.

```json
{
  "cuisine_profiles": {
    "korean": {
      "mains":      {"umami": 6, "salty": 4, "sweet": 8, "spicy": 9, "richness": 7},
      "appetizers": {"umami": 4, "salty": 5, "sweet": 3, "spicy": 6, "richness": 3},
      "desserts":   {"umami": 1, "salty": 2, "sweet": 9, "spicy": 1, "richness": 5}
    },
    "italian": {
      "mains": {"umami": 9, "salty": 7, "sweet": 2, "richness": 8}
    }
  },
  "common_liked_ingredients":    {"egg": 0.8, "tamarind": 0.6},
  "common_disliked_ingredients": {"cilantro": 0.9, "cream": 0.4},
  "dietary_restrictions":        ["no shellfish"]
}
```

`common_liked_ingredients` and `common_disliked_ingredients` are **frequency maps** derived from the user's rating history — how often an ingredient appears across dishes rated ≥ 7 or ≤ 4 respectively.

### Profile Lookup Fallback Hierarchy

When scoring a dish, the preference vector is resolved in this order:

1. `cuisine + section` profile — if ≥ 3 ratings exist for that combination
2. `cuisine`-only profile — averaged across all sections for that cuisine
3. Global profile — averaged across all cuisines and sections

### Profile Updates After Each Rating

After a user rates a dish, the relevant cuisine × section profile is nudged toward or away from the dish's flavor vector:

```python
learning_rate = 0.1
direction = +1 if dish_rating >= 7 else -1 if dish_rating <= 4 else 0

if direction != 0:
    for flavor in flavor_dimensions:
        profile[cuisine][section][flavor] += (
            direction * learning_rate
            * (dish.flavor_vector[flavor] - profile[cuisine][section][flavor])
        )
```

Ingredients from the rated dish are also added to the liked or disliked frequency maps accordingly.

## Scoring Function

### Step 1 — Hard Filters

Applied before any numeric scoring.

```
if dietary restriction violated → score = 0, exclude entirely
```

### Step 2 — Flavor Similarity (weight: 0.70)

Cosine similarity between the dish's flavor vector and the user's resolved preference vector for that cuisine × section combination (via fallback hierarchy above). Scaled by confidence as described in the sourcing section.

```python
flavor_score = cosine_similarity(
    dish.flavor_vector,
    user.cuisine_profiles[cuisine][section]
)
```

### Step 3 — Ingredient Score (weight: 0.30)

Measures overlap between the dish's ingredients and the user's liked/disliked ingredient history. Uses frequency maps so commonly liked ingredients contribute more than rare ones.

```python
boost   = sum(liked_ingredients.get(ing, 0)    for ing in dish.ingredients)
penalty = sum(disliked_ingredients.get(ing, 0) for ing in dish.ingredients)

ingredient_score = tanh(boost - penalty)  # bounded -1.0 to +1.0
```

For new users with no rating history, `ingredient_score` is neutral (0), so the flavor score carries the full recommendation weight.

### Step 4 — Final Score

```
score = (flavor_score     * 0.70
       + ingredient_score * 0.30)
```

## Match Level Thresholds

Match levels are derived from the user's own mean dish rating — their personal bar — rather than fixed global thresholds. This means a user who rates everything highly has a higher bar for "great" than a more critical rater.

```python
great_cutoff = (avg_dish_rating / 10) + 0.15
good_cutoff  = (avg_dish_rating / 10)

if score >= great_cutoff:   match_level = "great"
elif score >= good_cutoff:  match_level = "good"
else:                       match_level = "skip"
```

## Cold Start Handling

| User state | Flavor score source | Ingredient score |
|---|---|---|
| Brand new | Onboarding survey seeds cuisine profiles directly | Neutral (0) |
| Some history, cuisine known | Cuisine-only profile (section fallback) | Partial signal |
| Established history | Cuisine × section profile | Full signal |

Onboarding asks users to rate their preferred flavor intensities (1–10) for each cuisine they know, directly populating `cuisine_profiles` before any dishes have been scanned. This ensures personalization is meaningful from the very first recommendation.

---

## Summary of Signals Used

| Signal | Weight | Source |
|---|---|---|
| Flavor vector similarity (per cuisine × section) | 0.70 | Dish global entry, scaled by confidence |
| Ingredient overlap with liked/disliked history | 0.30 | Accumulated from post-scan ratings |
| Dietary restriction violations | Hard zero | User profile, set at onboarding |
| Confidence fallback to cuisine affinity | Embedded in flavor score | Scales automatically with source trust |
